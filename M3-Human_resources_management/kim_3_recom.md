# Методические указания для преподавателя

## Назначение

Этот КИМ используется как образец оформления модульного практического задания в ФОС. Его можно применять как текущий контроль по модулю 3 или как часть комплексного КИМ, связанного с планированием ресурсов проекта.

## Рекомендации по проведению

1. До занятия рекомендуется выдать студентам типовые описания ролей в AI-командах (Data Scientist, ML Engineer, Data Engineer, MLOps Engineer и т.д.).
2. На занятии целесообразно сначала разобрать понятие матрицы RACI на простом примере, не связанном с ИИ, а затем перейти к AI-кейсу.
3. Проектирование организационной структуры команды можно выполнить совместно с преподавателем.
4. Построение RACI-матрицы и анализ конфликтов рекомендуется выполнять самостоятельно или в малых группах.
5. Блок про лидерство и самоорганизацию удобно провести в формате дискуссии с опорой на модель Такмана.

## Что проверять в первую очередь

- соответствует ли структура команды масштабу и специфике выбранного AI-кейса;
- нет ли противоречий в матрице RACI (например, несколько Accountable на одном этапе, отсутствие Responsible);
- конкретны ли описанные конфликты интересов или они носят общий характер;
- связаны ли рассуждения о лидерстве с этапами развития команды;
- описано ли реальное кросс-функциональное взаимодействие, а не общие фразы о «важности коммуникации».

## Типичные ошибки студентов

1. Копируют типовой список ролей без адаптации к конкретному кейсу.
2. Указывают более одного «Accountable» для одного этапа в RACI.
3. Ограничиваются перечислением конфликтов без предложения способов их разрешения.
4. Описывают лидерство абстрактно, без привязки к этапам развития команды.
5. Не разграничивают роли Data Science и Engineering команд при описании кросс-функционального взаимодействия.
6. Не прикладывают визуализацию организационной структуры.

## Минимальная версия задания

Для слабой группы можно сократить задание до:

1. описания кейса и ролей команды;
2. построения RACI-матрицы для 3–4 ключевых этапов;
3. анализа одного конфликта интересов;
4. краткого описания принципов лидерства.

## Расширенная версия задания

Для сильной группы можно добавить:

1. сравнение структуры команды для пилотного и промышленного проекта;
2. проектирование команды для распределённого (удалённого) формата работы;
3. разработку метрик эффективности командной работы;
4. анализ изменения состава команды при переходе от MVP к продакшн-версии.

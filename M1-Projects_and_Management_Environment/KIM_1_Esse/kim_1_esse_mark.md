# Рубрика оценивания аналитического эссе на тему "Эволюция проектного управления"

**Оценивание КИМ_1_Esse**

Оценивание проводится по единой 10-балльной шкале ФОС.

---

### 1. Критерии оценивания

| Критерий | Предмет оценивания | Баллы |
|----------|---------------------|-------|
| Определение проекта и продукта в контексте двух эпох | Четко сформулированы различия в природе проектов (индустриальный vs ИИ-проект), даны корректные определения. | 1 |
| Анализ трансформации «Железного треугольника». | Описаны изменения ограничений (Сроки → Бюджет → Содержание) при работе с ИИ. Выявлен новый критический элемент управления (качество данных / дрейф модели). | 1.5 |
| Сравнение жизненных циклов (Waterfall vs Agile/CRISP-DM). | Обоснована итеративность и цикличность ИИ-проекта (Data → Model → Deployment → Monitor). Показаны причины неприменимости каскадной модели. | 1.5 |
| Сравнение документации | Детально сопоставлены классические документы (Устав, План управления) и ИИ-специфические (Dataset Card, Model Card, MLOps Pipeline). Раскрыто содержание каждого документа. | 2.0 |
| Анализ команды и ролей | Сравнены роли в классическом и ИИ-проекте (Архитектор/Строитель vs Data Scientist, ML Engineer, Data Labeler). Описана новая зона ответственности PM. | 1.0 |
| Трансформация управления под давлением «черных ящиков» | Сделан обоснованный вывод о том, как меняется системная модель управления, роль PM и организационная структура | 1.5 |
| Аргументация реальными примерами | Приведены примеры реальных ИИ-систем (ChatGPT, автопилоты, системы распознавания и др.) в соответствии с вариантом из таблицы | 1.0 |
| Структура, логичность, грамотность | Наличие всех обязательных разделов (Введение, Основная часть, Заключение). Логичность изложения, профессиональная терминология, отсутствие стилистических и грамматических ошибок | 0.5 |
| **Итого** | | **10.0** |

### 2. Уровни достижения результата

| Уровень | Баллы | Описание |
|---------|-------|----------|
| Не достигнут | 0–3 | Задание не выполнено или выполнено формально. Отсутствует сравнение классического и ИИ-проектов. Не раскрыта специфика документации и жизненного цикла ИИ. Выводы отсутствуют или не соответствуют теме. Примеры не приведены. |
| Базовый | 4–5 | Даны общие определения проекта и продукта. Проведено поверхностное сравнение «Железного треугольника». Упомянуты различия в документации, но без детализации. Жизненный цикл описан фрагментарно. Приведён 1–2 примера без привязки к варианту. Структура соблюдена частично. |
| Средний | 6–7 | Корректно определены различия между эпохами. Описана трансформация «Железного треугольника» с выделением нового элемента. Проведено сравнение Waterfall и Agile/CRISP-DM с обоснованием итеративности. Перечислены ИИ-документы с кратким описанием. Сравнены роли в команде. Приведены примеры в соответствии с вариантом. Выводы присутствуют, но недостаточно глубоки. |
| Высокий | 8–9 | Выполнен полный объём работы по всем разделам. Проведён детальный анализ трансформации ограничений с указанием причин. Раскрыто содержание Dataset Card, Model Card, MLOps Pipeline. Описаны причины цикличности жизненного цикла ИИ. Проведён глубокий анализ изменения роли PM и организационной структуры. Примеры реальных ИИ-систем детализированы и привязаны к варианту. Выводы обоснованы и соответствуют проведённому анализу. Структура безупречна. |
| Продвинутый | 10 | Работа носит исследовательский характер. Предложены собственные классификации или модели трансформации. Проведён критический анализ ограничений «Железного треугольника» с предложением альтернативных метрик управления ИИ-проектами. Рассмотрены этические аспекты и проблемы «чёрного ящика» с предложением способов их решения. Выводы выходят за рамки шаблонных формулировок, содержат рекомендации для практики управления ИИ-проектами. |

### 3. Основания для снижения баллов

- Отсутствует сравнение классического и ИИ-проектов по одному из обязательных разделов (ограничения, жизненный цикл, документация, команда).
- «Железный треугольник» описан без учёта специфики ИИ (не выделено новое критическое измерение).
- Жизненный цикл ИИ описан как линейный (Waterfall) без обоснования итеративности и цикличности.
- Документация ИИ перечислена без раскрытия содержания (например, «в ИИ есть Model Card» без пояснения, что туда входит).
- Не указана роль AI Ethicist или этического аудита в ИИ-проекте.
- Примеры не приведены или носят общий характер без привязки к варианту из таблицы.
- Не сделан вывод о трансформации роли Project Manager в условиях работы с «чёрным ящиком».
- Выводы противоречат содержанию основной части или отсутствуют.
- Нарушена структура эссе (отсутствует введение или заключение).
- Текст содержит грубые фактические ошибки (например, путаница между ML Engineer и Data Scientist в зонах ответственности).
- Работа носит реферативный характер без собственного анализа и обобщений.

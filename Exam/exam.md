# ТРЕБОВАНИЯ К ВЫПОЛНЕНИЮ, ОФОРМЛЕНИЮ И ЗАЩИТЕ СКВОЗНОГО ПРОЕКТА ПО ДИСЦИПЛИНЕ «УПРАВЛЕНИЕ ПРОЕКТАМИ В СФЕРЕ ИСКУССТВЕННОГО ИНТЕЛЛЕКТА»

## 1. Общие положения

### 1.1. Назначение документа
Настоящие требования определяют единый порядок выполнения, оформления, представления и защиты итогового проекта по дисциплине «Управление проектами в сфере искусственного интеллекта».

Итоговый проект является основной формой проверки сформированности профессиональных компетенций обучающихся и представляет собой комплекс взаимосвязанных документов по управлению AI-проектом, создаваемых в течение всего периода изучения дисциплины.

Проект выполняется поэтапно. Каждый этап соответствует тематике очередной лабораторной работы и используется в дальнейшем при разработке последующих разделов проекта. Таким образом, лабораторные работы имеют сквозной характер и формируют единый комплект проектной документации, который является предметом итоговой защиты.

### 1.2. Цель проекта
Цель выполнения проекта заключается в формировании практических навыков управления жизненным циклом AI-проекта посредством последовательной разработки основных документов проектного управления.

В процессе выполнения проекта обучающийся должен продемонстрировать способность:
1. анализировать предметную область;
2. формировать содержание проекта;
3. организовывать проектную команду;
4. планировать сроки выполнения проекта;
5. рассчитывать бюджет проекта;
6. управлять качеством;
7. проводить анализ рисков;
8. выбирать методологию управления;
9. использовать современные инструменты управления AI-проектами.

### 1.3. Принципы выполнения проекта
Проект строится на следующих принципах:
1. единая тема проекта на протяжении семестра;
2. последовательное развитие проекта;
3. использование результатов предыдущих лабораторных работ;
4. взаимосвязь всех разделов проекта;
5. единая структура документации;
6. обязательная самостоятельная защита каждым обучающимся своей части проекта.



## 2. Организация выполнения проекта

### 2.1. Сквозной характер проекта
Итоговый проект формируется постепенно. Каждая лабораторная работа является отдельным этапом разработки проекта и содержит результаты, которые используются при выполнении следующих лабораторных работ. Допускается корректировка ранее разработанных материалов при условии сохранения общей логики проекта.

Последовательность формирования проекта приведена в таблице.

| Этап | Лабораторная работа | Формируемый результат |
| --- | --- | --- |
| 1 | Устав AI-проекта | Паспорт проекта, цели, требования |
| 2 | Управление содержанием | WBS, содержание проекта |
| 3 | Формирование AI-команды | RACI, команда, перечень работ |
| 4 | Планирование сроков | Сетевой график, диаграмма Ганта |
| 5 | Управление стоимостью | Бюджет проекта |
| 6 | Управление качеством | План качества |
| 7 | Управление рисками | Реестр рисков |
| 8 | Выбор методологии | План реализации проекта |
| Экзамен | Защита проекта | Итоговая презентация |

### 2.2. Требования к самостоятельности
Обучающийся обязан самостоятельно:
1. выполнять расчеты;
2. принимать проектные решения;
3. обосновывать выбранные методы;
4. анализировать результаты;
5. представлять результаты защиты.

Использование генеративного искусственного интеллекта допускается исключительно как вспомогательного инструмента. Ответственность за достоверность материалов полностью несет обучающийся.



## 3. Программные средства
Рекомендуемые инструменты:
1. Google Docs / MS Excel — для построения RACI-матрицы;
2. Miro / Trello / MS Visio (или аналог) — для визуализации организационной структуры команды;
3. текстовый редактор — для аналитической записки;
4. MS Project / GanttProject — для построения диаграммы Ганта;
5. MS Excel / Google Sheets — для расчёта сетевой модели и критического пути вручную;
6. при необходимости — Python (например, библиотека networkx) для автоматизации расчёта критического пути;
7. Microsoft PowerPoint / LibreOffice Impress / Google Slides — для создания презентации.

Допускается использование других инструментов визуализации и планирования.



## 4. Структура итогового проекта
Итоговый проект должен содержать следующие разделы.

* Титульный лист
* Содержание
* **Введение** (актуальность, цель, задачи, объект, предмет, ожидаемый результат)
* **Раздел 1. Общая характеристика проекта** (описание проблемы, обоснование применения ИИ, заинтересованные стороны, ограничения, критерии успешности)
* **Раздел 2. Управление содержанием проекта** (устав проекта, WBS, описание результатов, границы проекта)
* **Раздел 3. Организация проектной команды** (организационная структура, RACI, распределение ролей, коммуникации)
* **Раздел 4. Планирование проекта** (перечень работ, сетевая модель, критический путь, диаграмма Ганта)
* **Раздел 5. Управление стоимостью** (бюджет, расчет стоимости, анализ отклонений, экономическое обоснование)
* **Раздел 6. Управление качеством** (показатели качества, критерии приемки, план контроля качества)
* **Раздел 7. Управление рисками** (реестр рисков, вероятности, последствия, мероприятия реагирования)
* **Раздел 8. Методология управления** (обоснование методологии, описание процесса, план спринтов)
* **Заключение** (достигнутые результаты, ограничения, перспективы)
* **Приложения** (диаграммы, таблицы, схемы, расчеты, доп. материалы)



## 5. Требования к оформлению

### 5.1. Общие требования
Итоговый проект представляется в виде единого текстового документа, содержащего результаты выполнения всех лабораторных работ, объединённых в единый проект по управлению проектом в сфере искусственного интеллекта. Документ должен быть логически завершённым, структурированным и соответствовать требованиям к оформлению учебной и научно-технической документации.

Текст проекта оформляется с использованием текстового редактора Microsoft Word или совместимого программного обеспечения и предоставляется в электронном виде в формате PDF и (при необходимости) DOCX. Все разделы проекта должны иметь единую структуру, стиль оформления и сквозную нумерацию.

### 5.2. Формат документа
Проект оформляется на листах бумаги формата А4 (210 × 297 мм) с использованием только одной стороны листа. Параметры страницы:
1. левое поле — 30 мм;
2. правое поле — 15 мм;
3. верхнее поле — 20 мм;
4. нижнее поле — 20 мм.

Выравнивание текста осуществляется по ширине страницы.

### 5.3. Шрифт и параметры текста
* шрифт — Times New Roman;
* размер шрифта — 14 pt;
* межстрочный интервал — полуторный (1,5);
* цвет шрифта — чёрный;
* начертание — обычное.

Абзацный отступ составляет 1,25 см. Перед и после абзацев дополнительные интервалы не устанавливаются. Не допускается использование различных гарнитур шрифтов, декоративного оформления текста, цветного выделения основного текста и неоправданного применения полужирного начертания. Для смыслового выделения допускается использование курсивного начертания.

### 5.4. Заголовки разделов и подразделов
Структурные элементы проекта («СОДЕРЖАНИЕ», «ВВЕДЕНИЕ», «ЗАКЛЮЧЕНИЕ», «СПИСОК ИСПОЛЬЗОВАННЫХ ИСТОЧНИКОВ», «ПРИЛОЖЕНИЯ») располагаются с новой страницы. Разделы нумеруются арабскими цифрами (например: `1 Общая характеристика проекта`). Подразделы имеют составную нумерацию (например: `1.1 Цель проекта`). Пункты подразделов: `1.2.1`, `1.2.2`.

Заголовки разделов печатаются полужирным шрифтом без точки в конце. Перенос слов в заголовках не допускается. Подчёркивание заголовков не применяется. Между заголовком и последующим текстом оставляется одна пустая строка.

### 5.5. Нумерация страниц
Все страницы проекта подлежат сквозной нумерации арабскими цифрами. Титульный лист включается в общую нумерацию, однако номер на нём не проставляется. Начиная со второй страницы номер располагается в нижней части страницы по центру.

### 5.6. Перечисления
* Если элементы списка не требуют указания последовательности — маркированный список.
* Если последовательность важна — нумерованный список.
* Каждый пункт начинается со строчной буквы и заканчивается точкой с запятой, кроме последнего (точка).

### 5.7. Оформление таблиц
Все таблицы должны иметь порядковый номер и наименование (например: *Таблица 4 — План управления рисками проекта*). Наименование располагается над таблицей слева. На все таблицы должны быть ссылки в тексте. При переносе указывается *Продолжение таблицы 4* или *Окончание таблицы 4*. Не допускается разделение заголовков строк и столбцов переносами слов.

### 5.8. Оформление рисунков
К рисункам относятся схемы, диаграммы, графики, блок-схемы, организационные структуры, диаграммы Ганта, сетевые графики, архитектуры AI-систем, иллюстрации. Подпись располагается под рисунком (например: *Рисунок 5 — Диаграмма Ганта проекта*). На каждый рисунок должна быть ссылка в тексте.

### 5.9. Оформление формул
Формулы располагаются по центру строки. Выше и ниже — одна свободная строка. Нумеруются арабскими цифрами в круглых скобках у правого края. После формулы приводится расшифровка обозначений.

### 5.10. Оформление ссылок
В тексте используются ссылки в квадратных скобках (например: `[5]` или `[3, 7, 12]`). Все источники должны присутствовать в списке использованных источников.

### 5.11. Список использованных источников
Размещается после заключения. Источники располагаются в порядке появления ссылок либо в алфавитном порядке. Оформление по ГОСТ Р 7.0.100–2018. Не рекомендуется использовать непроверенные интернет-ресурсы и материалы без авторства.

### 5.12. Оформление приложений
Каждое приложение начинается с новой страницы и обозначается заглавными буквами русского алфавита (например: *Приложение А. Диаграмма Ганта проекта*). В основном тексте на каждое приложение должна быть ссылка.

### 5.13. Требования к качеству оформления
* единый стиль оформления;
* отсутствие ошибок;
* корректная структура и автоматическое содержание;
* единая система нумерации;
* логическая взаимосвязь элементов.

**Не допускаются:** отсутствие ссылок на таблицы/рисунки/приложения, использование различных шрифтов без необходимости, нарушение структуры, отсутствие обязательных разделов, дублирование материалов, нечитаемые изображения, наличие неотредактированных сгенерированных ИИ фрагментов.



## 6. Представление проекта
К защите обучающийся предоставляет:
1. итоговый отчет;
2. презентацию;
3. комплект лабораторных материалов;
4. при необходимости ссылку на прототип;
5. журнал применения ИИ.



## 7. Порядок защиты
Защита проекта проводится в форме командной презентации (питча).
Продолжительность защиты:
* презентация проекта — 10–15 минут;
* ответы на вопросы комиссии — до 10 минут.

Во время защиты обучающиеся должны:
1. представить проблему и цель проекта;
2. обосновать применение технологий ИИ;
3. показать структуру команды и распределение ролей;
4. представить календарный план проекта;
5. обосновать расчёт сроков и бюджета;
6. представить мероприятия по обеспечению качества;
7. представить анализ рисков проекта;
8. обосновать выбор методологии управления;
9. продемонстрировать результаты выполнения проекта;
10. ответить на вопросы комиссии.

Каждый участник команды обязан представить результаты своей части проекта и подтвердить личный вклад. При наличии сомнений комиссия вправе задать индивидуальные вопросы любому участнику.

## 9. Уровни оценивания
* **Отлично (9–10)** — Обучающийся демонстрирует сформированность компетенций на высоком уровне. Все документы логически согласованы, расчеты корректны, проект реализуем, ответы полные и аргументированные.
* **Хорошо (7,5–8,9)** — Допускаются отдельные неточности, не влияющие на общую работоспособность проекта.
* **Удовлетворительно (6–7,4)** — Проект выполнен полностью, однако имеются методические ошибки, недостаточная глубина анализа и неполное обоснование решений.
* **Неудовлетворительно (менее 6)** — Проект содержит существенные ошибки, отсутствуют обязательные разделы, обучающийся не может обосновать принятые решения.

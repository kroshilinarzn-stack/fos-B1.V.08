# КИМ 8. Лабораторная работа. Планирование спринта на основе методологии Lean DS

## 1. Паспорт работы

| Параметр | Значение |
|---|---|
| Дисциплина | Управление проектами |
| Тема | Планирование спринта на основе методологии Lean DS |
| Вид оценочного средства | Лабораторная работа |
| Вид контроля | Текущий контроль |
| Рекомендуемый формат выполнения | Индивидуально или в составе AI-команды |
| Рекомендуемая среда | Yandex Board / Holst / MS Excel |
| Основной артефакт | Бэклог спринта  + декомпозиция продуктовых гипотез методом “Мерседес” + краткий отчёт |
| Рекомендуемая трудоёмкость | 2 академических часа аудиторной работы + самостоятельная подготовка расчётов и отчёта |

## 2. Назначение работы

Работа предназначена для проверки того, что обучающийся умеет декомпозировать продуктовые гипотезы методом “Мерседес”.

Задание также проверяет умение приоритизировать задачи и выстраивать программу регулярных встреч команды.

## 3. Проверяемый результат обучения

Обучающийся декомпозирует продуктовые гипотезы AI-проекта методом “Мерседес” и формирует приоритизированный бэклог релиза. Формирует расприсание и план проведения регулярных встреч команды.

## 4. Проверяемые результаты и компетенции

| Компетенция / область результата | Ориентировочный индикатор | Уровень | Как проявляется в задании |
|---|---|---|---|
| LC-4 | 4.1 | Продвинутый | Декомпозиция целей проекта и организация регулярных встреч команды |

## 5. Исходные данные

Для выполнения задания используется учебный кейс AI-проекта (может совпадать с кейсом из предыдущих модулей), содержащий:

1. краткое описание AI-продукта и требований к его качеству со стороны заказчика/пользователей;
2. перечень требований к технологиям ИИ в проекте;
3. сведения о составе команды.

## 6. Программные средства

Рекомендуемый стек:

- Yandex Board;
- Holst;
- MS Excel.

Допускается использовать иные инструменты построения матрицы и реестра рисков.

## 7. Ход выполнения

### Часть 1. Анализ исходных данных и постановка цели спринта

**Цель:** определить границы спринта, сформулировать измеримую цель и согласовать ожидаемые результаты с заказчиком.

**Действия:**

1. Внимательно изучите описание ИИ-продукта, требования к качеству и состав команды.

2. Сформулируйте цель предстоящего спринта. Цель должна быть конкретной, измеримой, достижимой, значимой и ограниченной во времени. Например: «Разработать и протестировать MVP модели кредитного скоринга с точностью не менее 80 % на валидационной выборке».

3. Определите, какой результат должен быть получен по итогам спринта: рабочая версия модели, отчёт о качестве данных, прототип API или иной артефакт.

4. Согласуйте цель спринта с ожиданиями заказчика и ключевых стейкхолеров.

**Результат части 1:** сформулированная и согласованная цель спринта.

---

### Часть 2. Декомпозиция цели на продуктовые гипотезы

**Цель:** разбить цель спринта на проверяемые гипотезы, которые помогут последовательно приближаться к результату.

**Действия:**

1. Представьте, что цель спринта — это главный вопрос, на который нужно ответить к его завершению.

2. Сформулируйте продуктовые гипотезы — утверждения о том, какие действия приведут к достижению цели. Каждая гипотеза должна быть проверяемой и иметь чёткий критерий успеха.

3. Для каждой гипотезы определите:
   - что нужно сделать для её проверки;
   - какой результат подтвердит или опровергнет гипотезу;
   - как этот результат будет измеряться.

4. Заполните таблицу гипотез.

**Результат части 2:** список продуктовых гипотез с критериями проверки.

---

### Часть 3. Декомпозиция гипотез методом «Мерседес»

**Цель:** разложить каждую продуктовую гипотезу на отдельные задачи, достаточные для планирования спринта.

**Действия:**

1. Метод «Мерседес» предполагает декомпозицию по трём уровням:
   - **Уровень 1. Продуктовая гипотеза** (что проверяем).
   - **Уровень 2. Эксперимент или активность** (как проверяем).
   - **Уровень 3. Конкретные задачи** (что делаем по шагам).

2. Для каждой гипотезы (уровень 1) опишите, какие эксперименты или активности потребуется провести для её проверки (уровень 2).

3. Каждую активность разбейте на отдельные задачи (уровень 3). Задачи должны быть достаточно мелкими, чтобы их можно было оценить в часах и выполнить в рамках спринта.

4. Для каждой задачи укажите:
   - краткое описание;
   - исполнителя или роль;
   - трудоёмкость в часах или стори-поинтах;
   - зависимость от других задач.

5. Проверьте полноту декомпозиции: покрывают ли задачи все этапы проверки гипотезы.

**Результат части 3:** полная декомпозиция гипотез на задачи с указанием исполнителей и трудоёмкости.

---

### Часть 4. Формирование бэклога спринта

**Цель:** структурировать все задачи в единый список и организовать их для дальнейшей приоритизации.

**Действия:**

1. Объедините все задачи из декомпозиции в единый список — бэклог спринта.

2. Для каждой задачи укажите:
   - уникальный номер;
   - название или краткое описание;
   - связь с гипотезой (какую гипотезу проверяет);
   - исполнителя или роль;
   - трудоёмкость;
   - зависимости от других задач.

3. Оцените суммарную трудоёмкость всех задач. Сравните с доступным временем команды в спринте.

4. При необходимости сократите объём задач, исключив наименее приоритетные.

**Результат части 4:** полный бэклог спринта с оценкой трудоёмкости.

---

### Часть 5. Приоритизация задач

**Цель:** расставить задачи по степени важности для достижения цели спринта.

**Действия:**

1. Используйте один из методов приоритизации:
   - **MoSCoW:** разделите задачи на Must have, Should have, Could have, Won't have.
   - **Приоритет по риску:** сначала выполняются задачи, связанные с наибольшими рисками для проекта.
   - **Приоритет по ценности:** сначала выполняются задачи, приносящие наибольшую ценность заказчику.

2. Для каждой задачи определите её приоритет и обоснуйте выбор.

3. Упорядочите бэклог по убыванию приоритета. В начале спринта команда должна выполнять наиболее важные задачи.

4. Проверьте, что задачи с наивысшим приоритетом позволяют достичь цели спринта даже в случае, если менее приоритетные задачи не будут выполнены.

**Результат части 5:** приоритизированный бэклог спринта с обоснованием приоритетов.

---

### Часть 6. Распределение задач по исполнителям

**Цель:** закрепить задачи за конкретными участниками команды с учётом их ролей и компетенций.

**Действия:**

1. Сопоставьте задачи с ролями участников команды (Data Scientist, Data Engineer, ML Engineer, DevOps, Project Manager и другие).

2. Убедитесь, что нагрузка распределена равномерно. Если один исполнитель перегружен, рассмотрите возможность перераспределения задач между участниками с близкими компетенциями.

3. Оцените загрузку каждого исполнителя в часах. Она не должна превышать доступное время в спринте (с учётом совещаний и непредвиденных работ).

4. При необходимости скорректируйте бэклог: перенесите часть задач в следующий спринт или пересмотрите их трудоёмкость.

**Результат части 6:** распределение задач по исполнителям с оценкой индивидуальной загрузки.

---

### Часть 7. Планирование регулярных встреч команды

**Цель:** составить расписание встреч, обеспечивающих синхронизацию команды и контроль прогресса.

**Действия:**

1. Определите, какие типы встреч необходимы в рамках выбранной методологии Lean DS. Рекомендуемый набор включает:
   - **Ежедневный стендап** (15 минут) — синхронизация текущего статуса задач.
   - **Встреча по планированию спринта** (1–2 часа) — в начале спринта.
   - **Встреча по ретроспективе** (1 час) — в конце спринта.
   - **Демонстрация результатов** (1 час) — по итогам спринта для заказчика.
   - **Встреча по уточнению требований** (по необходимости) — для работы с бэклогом.

2. Для каждой встречи укажите:
   - тип и цель встречи;
   - продолжительность;
   - периодичность (ежедневно, еженедельно, по итогам спринта);
   - участники;
   - формат проведения (очный, онлайн).

3. Заполните календарный план встреч на период спринта.

**Результат части 7:** расписание регулярных встреч команды с указанием целей и участников.

---

### Часть 8. Подготовка итогового отчёта

**Цель:** представить результаты работы в виде связного документа.

**Действия:**

1. Сформулируйте цель спринта и ключевые продуктыые гипотезы.

2. Приведите таблицу декомпозиции гипотез методом «Мерседес» с уровнями декомпозиции.

3. Представьте приоритизированный бэклог спринта со всеми задачами.

4. Приведите расписание регулярных встреч команды.

5. Сделайте вывод о готовности команды к началу спринта и возможных рисках.

6. При необходимости включите в отчёт анализ загрузки исполнителей.

**Результат части 8:** итоговый отчёт, готовый к защите.

---

### Важные замечания

1. Все гипотезы должны быть проверяемыми. Если гипотезу нельзя подтвердить или опровергнуть за время спринта, её следует переформулировать.

2. При декомпозиции методом «Мерседес» следите, чтобы задачи на третьем уровне были достаточно мелкими — не более 8–16 часов на одну задачу.

3. Приоритизация должна учитывать не только ценность задачи, но и её риски: задачи с высокой неопределённостью рекомендуется выполнять раньше.

4. Расписание встреч должно быть реалистичным и учитывать, что команда работает в одном часовом поясе (или с учётом разницы во времени).

5. Проверьте, что суммарная трудоёмкость задач соответствует доступному времени команды. Если превышает — сократите бэклог.

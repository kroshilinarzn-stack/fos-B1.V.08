# КИМ для преддипломной практики
## «Управление проектами в сфере ИИ»

---

## МЕТОДИЧЕСКИЕ УКАЗАНИЯ ДЛЯ ПРЕПОДАВАТЕЛЯ

### Назначение КИМ

Контрольно‑измерительный материал предназначен для оценки уровня сформированности компетенций студентов в ходе **преддипломной практики**. Задания ориентированы на углублённый анализ и систематизацию материалов для выпускной квалификационной работы, с акцентом на экономическую эффективность, управление рисками, качество данных и модели, а также этические аспекты ИИ‑проектов.

### Основные функции КИМ

| **Функция** | **Описание** |
|-------------|--------------|
| Диагностическая | Выявление способности студента проводить комплексный анализ ИИ‑проекта по всем ключевым аспектам управления |
| Обучающая | В процессе выполнения задания студент углубляет знания о методах оценки экономической эффективности, управления рисками и качеством в ИИ‑проектах |
| Формирующая | Развивает навыки системного анализа, научно‑исследовательской работы и подготовки материалов для ВКР |
| Интегративная | Связывает знания из областей проектного менеджмента, экономики, Data Science и этики ИИ |
| Оценочная | Позволяет объективно оценить уровень сформированности компетенций LC‑1.3, LC‑4.1, LC‑4.2, LC‑7.1, LC‑7.3, SS‑2.1, SS‑2.2 |

### Рекомендации по проведению

1. Задание выдаётся в начале преддипломной практики; защита отчёта проводится в последнюю неделю.
2. Студент работает индивидуально над проектом, который ляжет в основу ВКР.
3. В качестве объекта анализа может выступать реальный ИИ‑проект организации или гипотетический проект, разрабатываемый студентом.
4. Критерии оценки доводятся до студентов до начала работы.
5. Рекомендуется провести установочную консультацию (30–40 минут) с разбором структуры аналитической записки и требований к каждому разделу.
6. Обязательная проверка на наличие заимствований и использование генеративных ИИ‑инструментов (допустимый порог оригинальности — не менее 70 %).
7. Каждая работа сопровождается развёрнутым комментарием с указанием сильных сторон и зон для улучшения.

### Что проверять

| **Знания** | **Умения** | **Навыки** |
|------------|------------|------------|
| Методы технико‑экономического обоснования ИИ‑решений | Строить расчётные модели для оценки экономических эффектов внедрения ИИ | Проведение TCO и ROI‑анализа ИИ‑проекта |
| Классификация и методы оценки рисков ИИ‑проектов | Выявлять и количественно оценивать риски, специфичные для ИИ | Разработка плана реагирования на риски |
| Стандарты качества и метрики ML‑моделей | Подбирать метрики качества для конкретной задачи ИИ | Разработка плана управления качеством AI‑продукта |
| Этические и правовые аспекты ИИ | Проводить этический и правовой аудит ИИ‑решения | Составление этического чек‑листа |
| Современные методологии управления (Agile, CRISP‑DM, MLOps) | Обосновывать выбор методологии для конкретного ИИ‑проекта | Адаптация гибких методов к условиям проекта |
| Методика освоенного объёма (EVM) | Применять EVM для прогнозирования бюджета и сроков | Расчёт показателей EVM по фактическим данным |

### Типичные ошибки студентов

1. **Поверхностный TCO‑анализ** — учтены только прямые затраты на разработку, без учёта эксплуатации и сопровождения.
2. **ROI рассчитан без учёта нематериальных выгод** — не учтены улучшение качества, скорость принятия решений и т.п.
3. **Риски описаны без количественной оценки** — отсутствует расчёт вероятности и влияния.
4. **Метрики качества выбраны формально** — без привязки к бизнес‑целям проекта.
5. **Этический аудит проведён поверхностно** — не рассмотрены вопросы предвзятости, объяснимости и конфиденциальности.
6. **Выбор методологии не обоснован** — студент выбирает Agile «по умолчанию», не анализируя специфику проекта.
7. **Отсутствие связи с реальными данными** — все расчёты выполнены на гипотетических цифрах.
8. **Выводы не соответствуют содержанию** — заключение не вытекает из основной части.
9. **Нарушение структуры** — пропущен один из обязательных разделов.
10. **Использование ИИ‑инструментов без указания** — не описано, какие инструменты применялись.

### Минимальная версия задания

Для студентов с низким уровнем подготовки или при ограниченном времени:

- Провести TCO‑анализ (без детальной разбивки по статьям затрат).
- Перечислить 5 ключевых рисков с качественной оценкой.
- Предложить 3 метрики качества для модели.
- Составить этический чек‑лист из 5 пунктов.
- Выбрать методологию управления с кратким обоснованием (1–2 предложения).

### 5. Практическое задание

На основе ИИ‑проекта, который вы планируете разрабатывать в рамках ВКР (или реального проекта организации), подготовьте аналитическую записку, содержащую следующие разделы:

1. **Технико‑экономическое обоснование (ТЭО) проекта**
   - Описание бизнес‑проблемы и обоснование целесообразности применения ИИ.
   - Расчёт совокупной стоимости владения (TCO) — затраты на разработку, внедрение, эксплуатацию и сопровождение.
   - Расчёт ожидаемого экономического эффекта (ROI, NPV, срок окупаемости).
   - Сравнение с альтернативными решениями (без ИИ, с использованием традиционных методов).

2. **Управление рисками**
   - Идентификация рисков, специфичных для вашего проекта (риски данных, модели, инфраструктуры, рыночные, этические, правовые).
   - Качественная и количественная оценка рисков (матрица вероятности и влияния).
   - Разработка плана реагирования на ключевые риски.

3. **Управление качеством AI‑продукта**
   - Определение метрик качества для ML‑модели (с учётом бизнес‑целей).
   - Разработка плана управления качеством (этапы контроля, критерии приемки, методы тестирования).
   - Применение диаграммы Парето для приоритизации задач качества.

4. **Этический и правовой аудит**
   - Оценка рисков предвзятости (bias) и дискриминации.
   - Анализ вопросов объяснимости (explainability) и интерпретируемости модели.
   - Проверка соответствия законодательству (GDPR, 152‑ФЗ и др.).
   - Разработка этического чек‑листа для проекта.

5. **Выбор методологии управления**
   - Обоснование выбора методологии (Agile, Scrum, Kanban, CRISP‑DM, MLOps или их комбинация) с учётом специфики проекта.
   - Описание адаптации выбранной методологии к условиям проекта.

6. **Заключение**
   - Обобщение результатов анализа.
   - Рекомендации по улучшению управления проектом.
   - План дальнейших действий для ВКР.

### 6. Формат сдачи

Обучающийся сдаёт **аналитическую записку** (отчёт по преддипломной практике), содержащую:

- титульный лист;
- задание на практику;
- основную часть (пп. 1–6);
- список использованных источников;
- приложения (расчётные таблицы, диаграммы, чек‑листы).

Объём: **20–30 страниц** (без учёта приложений).

### 7. Контрольные вопросы для защиты

1. Какова бизнес‑цель вашего ИИ‑проекта и как вы обосновали её достижимость?
2. Какие статьи затрат включены в TCO и почему?
3. Какой ожидаемый ROI и срок окупаемости? Какие допущения лежат в основе расчётов?
4. Какие риски вы оценили как наиболее критичные и почему?
5. Как вы планируете управлять рисками предвзятости и дискриминации в модели?
6. Какие метрики качества вы выбрали и почему они соответствуют бизнес‑целям?
7. Как вы планируете обеспечивать объяснимость модели для пользователей и регуляторов?
8. Почему вы выбрали именно эту методологию управления? Какие альтернативы рассматривали?
9. Как вы планируете организовать мониторинг модели после деплоя (data drift, concept drift)?
10. Какие этические принципы вы заложили в проект и как вы будете их соблюдать?

### 8. Использование ИИ‑инструментов

Разрешается использовать генеративные ИИ‑инструменты для:

- помощи в структурировании аналитической записки;
- генерации идей для ТЭО и планов управления;
- проверки текста на грамматические и стилистические ошибки.

**Обязательно указывается:**

- какой инструмент использовался;
- для какой части работы;
- как обучающийся убедился в корректности результата.

Не допускается сдача сгенерированной аналитической записки без критической интерпретации и понимания результата.

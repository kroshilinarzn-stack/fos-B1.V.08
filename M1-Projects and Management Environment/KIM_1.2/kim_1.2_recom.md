# КИМ_1.2. Эволюция проектного управления

## Методические указания для преподавателя

### Назначение КИМ

Контрольно-измерительный материал предназначен для текущей оценки уровня сформированности компетенций студентов в области понимания основ проектного управления и специфики управления ИИ-проектами. Его цель — обобщить и систематизировать знания, полученные студентами в ходе изучения теоретического материала (на лекциях и в процессе самостоятельной работы) основных понятий модуля 1 «Проекты и среда управления».

### Основные функции КИМ

| Функция | Описание |
|---------|----------|
| Диагностическая | Выявление уровня понимания студентом фундаментальных различий между управлением классическими (индустриальными) и ИИ-проектами |
| Обучающая | В процессе выполнения задания студент систематизирует знания о специфике ИИ-проектов (MLOps, документация, этика, управление данными) |
| Формирующая | Развивает навыки системного анализа, сравнительного исследования и письменной аргументации в профессиональной области |
| Интегративная | Связывает знания из областей проектного менеджмента, Data Science, этики ИИ и управления данными в единую картину |
| Оценочная | Позволяет объективно оценить уровень сформированности компетенций LC-1.1, LC-1.3 |

---

### Рекомендации по проведению

1. Рекомендуется выдавать задание не позднее чем за 2 недели до даты сдачи, чтобы студенты имели время для изучения источников и глубокого анализа.
2. Вариант может назначаться строго по номеру в списке группы либо случайным образом. Рекомендуется закрепить вариант за студентом, чтобы избежать повторяющихся работ.
3. Критерии оценки должны быть объявлены до начала выполнения задания, чтобы студенты понимали требования и ожидания.
4. Рекомендуется провести установочную консультацию (15–20 минут), на которой разъясняется структура эссе, ключевые термины и требования к источникам.
5. Обязательная проверка на наличие заимствований и сгенерированный текст. Допустимый порог − не менее 70%.
6. Каждая работа должна сопровождаться комментарием с указанием сильных сторон и зон для улучшения.

### Что проверять

| Знания | Умения | Навыки |
|--------|--------|--------|
| Определение проекта и продукта в классическом и ИИ-контексте | Сравнивать индустриальный и ИИ-проекты по ключевым управленческим параметрам | Аналитическое мышление и сравнительный анализ |
| Модель «Железного треугольника» и её ограничения | Выявлять трансформацию ограничений и адаптировать управленческие подходы | Работа с научной и профессиональной литературой |
| Каскадная (Waterfall) и гибкая (Agile/CRISP-DM) модели жизненного цикла | Аргументировать выбор итеративного жизненного цикла для ИИ-проекта | Структурированное письменное изложение |

### Типичные ошибки студентов

1. **Отсутствие сравнения** — описание классического и ИИ-проекта по отдельности без сопоставления
2. **«Железный треугольник» без специфики ИИ** — описан только классический треугольник, нет нового элемента
3. **Линейное описание жизненного цикла ИИ** — студент описывает ИИ-проект как последовательность этапов без цикличности
4. **Документация перечислена без раскрытия содержания** — студент просто перечисляет Dataset Card, Model Card, но не объясняет, что в них входит
5. **Путаница между Data Scientist и ML Engineer** — студент приписывает инженерные задачи Data Scientist и наоборот
6. **Отсутствие AI Ethicist** − студент не упоминает этические роли и риски
7. **Примеры не привязаны к варианту** — общие примеры из интернета, а не из выбранной пары
8. **Выводы не соответствуют содержанию** — заключение написано шаблонно и не вытекает из основной части
9. **Структура нарушена** — пропущен один из обязательных разделов или нарушена последовательность
10. **Реферативный характер** — работа представляет собой пересказ источников без собственного анализа

### Минимальная версия задания

Используется для студентов с низким уровнем подготовки, при ограниченном времени или в качестве допуска к основной работе.

| Раздел | Содержание |
|--------|------------|
| 1. Введение | Определить проект и продукт в классике и ИИ (кратко, 2–3 предложения на каждое определение). |
| 2. «Железный треугольник» | Описать классический треугольник + указать 1 новый элемент для ИИ (без подробного анализа). |
| 3. Жизненный цикл | Назвать Waterfall и CRISP-DM, указать, что ИИ-цикл итеративен (без развёрнутого обоснования). |
| 4. Документация | Перечислить 3 ИИ-документа (Dataset Card, Model Card, MLOps Pipeline) с кратким пояснением (1–2 предложения о каждом). |
| 5. Команда | Перечислить роли в ИИ-команде (Data Scientist, ML Engineer, Data Labeler) — кратко. |
| 6. Заключение | Один общий вывод о трансформации управления (2–3 предложения). |

---

### Расширенная версия задания

Используется для студентов с высоким уровнем подготовки, в том числе в рамках научно-исследовательской работы или дифференцированного зачёта.

| Раздел | Дополнительные требования |
|--------|---------------------------|
| 1. Введение | Сравнительный анализ определений проекта в стандартах PMI, ГОСТ Р 54869 и подходах MLOps. |
| 2. «Железный треугольник» | Предложить авторскую модель ограничений для ИИ-проекта (не менее 4 элементов с обоснованием связей). Сравнить с существующими моделями (например, «Железный треугольник 2.0»). |
| 3. Жизненный цикл | Подробное сравнение CRISP-DM, TDSP (Microsoft) и собственного предложения. Описание автоматических триггеров переобучения. |
| 4. Документация | Детальный разбор содержимого Dataset Card и Model Card с примерами для своего варианта. Добавить этический чек-лист (AI Ethics Checklist) как новый документ. |
| 5. Команда | Рассмотреть также роли Data Engineer, AI Product Manager, MLOps Engineer. Описать модель взаимодействия ролей (матрица RACI). |
| 6. Заключение | Разработать рекомендации по адаптации PM-стандартов (PMBOK, PRINCE2, Agile) для ИИ-проектов. Оценить риски и пути их минимизации. |
